## Answer review questions directly

When asked a clarifying or double-checking question mid-implementation, answer it directly and plainly. Don't frame the answer around how settled the plan is, how far along implementation has gotten, or whether the question was "necessary" — treat it as a normal part of ongoing code review, not friction to get past, regardless of how much work already happened before the question was asked.

## Surface scope mismatches

If your own implementation instincts imply scope beyond what was explicitly asked for or written in a ticket (a new capability, not just an implementation detail), raise that mismatch for discussion at the time — don't silently decide solo whether to keep the extra scope or trim back to the literal ask. Concrete trigger: if you catch yourself writing code that only makes sense if some capability exists, and that capability wasn't actually requested, stop and name the gap out loud before choosing to build it or cut it.

## Clean code, not just correct code

Code that works correctly can still have a genuine, articulable maintainability problem — mixed concerns, domain-specific logic sitting in a general-purpose file, a missing single-responsibility split — that will bite the next person who builds on it. When asked to assess whether code is good or where something belongs, "it's correct" is not the same bar as "it's clean" — a real maintainability issue deserves to be named, not waved off because nothing is currently broken.

## Tested inferior code is not a cost to preserve

When a better approach would require rewriting or deleting existing tests, the fact that the current approach is already tested is never a mark against the change. Rewriting tests to match improved code is part of the work, not a cost weighed against doing it — so don't list "the old behavior is tested" as friction, a downside, or a reason to hesitate, and don't let test coverage of an inferior approach anchor a decision toward keeping it. Tests exist to protect the behavior you want, not to entrench the implementation you happen to have. (This is the same principle as never letting existing code style justify keeping a worse pattern, applied to the test suite: precedent — including test precedent — only ever informs a decision after merit, never gates it.)

## Module organization

`app/` is organized by domain, not by technical type — there is no shared `models.py`, `schemas.py`, or `routers/` bucketing every domain's files by kind. `app/user/`, `app/goal/`, `app/profile/`, and `app/swim_time/` are the base entity domains, each owning `model.py` (SQLAlchemy) and `schema.py` (Pydantic) for its own entity; `goal`/`profile`/`swim_time` also own a `router.py` since each entity is managed end-to-end through its own endpoints. `app/user/` currently has no `router.py` of its own — there's no standalone "manage a user" HTTP surface yet, only the `auth` domain reading the `User` model and its `UserOut` schema.

`app/auth/`, `app/stats/`, and `app/rag/` are capability domains: named for what they do rather than for an entity they own, and each reads across one or more base domains rather than owning one itself. `app/auth/` has no `model.py` — it owns the authentication capability (`schema.py`: `UserCreate`/`UserLogin`/`Token`; `router.py`: register/login/logout/token issuance) and imports `User`/`UserOut` from `app/user/`. `app/stats/` is the same shape minus any model — a read-only reporting domain querying across `user`/`swim_time`. This mirrors DDD's point (Evans, *Domain-Driven Design*) that a bounded context is named after what the ubiquitous language actually calls it, not mechanically after whichever entity its model defines — don't assume a domain must be entity-named just because `goal`/`profile`/`swim_time` are.

The domain-vs-layer split itself (grouping a domain's own files together instead of bucketing by file kind) is a deliberate choice over a layered/hexagonal split (interface layer kept separate from domain logic): for a single-FastAPI-app CRUD service, a domain's model, schema, and router almost always change together for the same reason (the Common Closure Principle, Martin's *Agile Software Development*), so colocating them cuts navigation cost with no real flexibility given up.

Domain-owned enums follow the same rule: a `Literal` type used by only one domain lives in that domain (e.g. `app/goal/enums.py`, or inlined directly in the owning file if there's only one consumer, like the RAG-only taxonomy literals in `app/rag/models.py`). `app/enums.py` at the top level is reserved for the rare case a vocabulary is genuinely shared across domains — today just `StrokeLiteral`, used by both `swim_time` and `rag`. Don't add something to `app/enums.py` just because it feels "general"; check it's actually consumed by more than one domain first.

Tests are colocated with what they test, for the same reason as everything else above: `app/goal/router_test.py` sits next to `app/goal/router.py`, `app/rag/sharpen_test.py` next to `app/rag/sharpen.py`, and so on. Cross-cutting files that stay flat at the top of `app/` get their tests colocated the same way (`app/main_test.py`, `app/config_test.py`, `app/database_test.py`, `app/docs_test.py` next to `main.py`/`config.py`/`database.py`/`docs.py`). There is no separate `tests/` directory - `app/conftest.py` at the top of `app/` provides fixtures to every test under it, including nested domain folders. Test files use the `*_test.py` suffix rather than pytest's other default of `test_*.py`, since the suffix form sorts and scans next to its sibling file in a directory listing instead of collecting every test in the codebase under one `test_` bucket - pytest's default `python_files` setting (`test_*.py`, `*_test.py`) picks up both forms natively, so this is a pure naming choice with no discovery-config cost.

A test file's job is to verify its own sibling file's contract in isolation, never to lean on some caller already exercising it. A router test checks the router's own input/output; it should never be the only thing verifying the logic of whatever it calls. Concretely: hitting `/auth/me` with an expired token and getting a 401 back proves the *router* handles that response correctly, but it is not a substitute for `app/security_test.py` directly calling `decode_access_token()` with an expired token - those are different contracts, and "some router's test happens to walk through these lines" is never a reason to skip a file's own dedicated test. `app/domain_isolation_test.py` is the one exception to "test what's next to it" - it asserts that `app/user/`, `app/goal/`, `app/profile/`, and `app/swim_time/` never import from each other (see the module organization above), which is a property of the whole `app/` tree, not any single file.

`app/rag/` is one domain that's internally organized by pipeline step (`query.py`, `embeddings.py`, `retrieval.py`, `answer.py`, `sharpen.py`, `sharpen_flag.py`, `training.py`, plus its own `models.py`, `schema.py`, `router.py`) rather than the plain `model.py`/`schema.py`/`router.py` shape the thinner domains use — deliberate, because RAG has enough internal steps that each one is an independently-readable, independently-testable unit of the design, not because any step is reused elsewhere. "This only has one caller" is not a reason to inline something back into its caller if it represents a real conceptual unit; reserve colocation for genuinely incidental glue code with no identity of its own.

Cross-cutting technical concerns that aren't owned by any single domain stay flat at the top of `app/` instead of being forced into a domain folder: `database.py` (engine/session/declarative bases), `deps.py` (shared FastAPI dependencies), `security.py`, `rate_limit.py`, `docs.py`, `model_utils.py`, and the `*_config.py` settings modules. These are legitimately layer-first, not a leftover of the old type-bucket structure — a domain folder would be the wrong home for something every domain depends on.

For actual duplicated logic/constants (the same snippet copy-pasted, not a designed step), extract to a shared location once a real second duplicate appears — but use judgment rather than treating that as an automatic trigger: two superficially similar cases can be coincidentally similar rather than truly the same concept, and forcing a shared abstraction from only two data points is a common way to guess the wrong interface.
